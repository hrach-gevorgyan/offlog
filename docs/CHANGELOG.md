# Offlog — Changelog

The single source of truth for version history — this used to be duplicated
across `TECH.md` (technical detail) and the root `README.md` (short
highlights), which had drifted in coverage (TECH.md's table was more
complete). Consolidated here 2026-07 as part of the documentation cleanup;
both `README.md` and `docs/TECH.md` now link here instead of keeping their
own copy. New entries go at the top, one row per released version.

This table keeps the newest ~8 releases in full detail. Older releases are
compressed to one line each in [CHANGELOG-ARCHIVE.md](CHANGELOG-ARCHIVE.md)
(git still has every full diff) — split out 2026-07 once this file's
default-loaded size grew large enough to matter. When adding a new entry,
move the oldest row here into the archive (compressed to one line) so this
table stays roughly the same length release over release.

| **5.2.1** | **Two real bugs from owner testing on-device**: biometric unlock's toggle in Settings → App Lock was gated only on `isNativePlatform()`, not on a PIN actually being set (`SettingsPanel.svelte`) — biometric alone with no PIN makes no sense given B54's whole design (biometric sits *alongside* the PIN, never replaces it); now gated on `isNativePlatform() && appLockEnabled` so the section only appears once a PIN exists. **Old logo on the Android splash screen**: `resources/generate-icons.cjs` (the rebrand pipeline) regenerates the launcher/notification icons from `source-logo.svg` but never touched `drawable*/splash.png` — the legacy pre-API-31 splash fallback stayed on the old pre-rebrand mark. New `resources/generate-splash.cjs` regenerates all 26 density/orientation/night variants, matching `colorPrimaryDark` so the legacy fallback looks identical to the modern (API 31+) `windowSplashScreenBackground`/`windowSplashScreenAnimatedIcon` splash instead of just less-wrong. **Cleanup**: deleted a fully orphaned `offlog-app/assets/` directory — an old, superseded icon-generation pipeline (pre-dating `resources/`) still holding the actual old-logo source PNGs, confirmed unreferenced by any code, config, or doc before removal |
| **5.2.0** | **Privacy Screen + Clipboard (B55/B56)** — two small plugins picked up from a Capacitor-plugin survey after biometric unlock shipped. **Privacy Screen**: closes a gap App Lock's PIN alone couldn't — Android snapshots whatever's on screen the instant the app backgrounds, before the lock screen even covers it, so a PIN-locked app could still leak a full task-list preview in the recent-apps switcher. `config.ts`'s new `syncPrivacyScreen()` dims that preview whenever a PIN is set, called at launch and again on every PIN set/remove so it can't drift out of sync mid-session; best-effort (never blocks using the app if the plugin fails). **Clipboard**: a "Copy" button next to the App Lock recovery code's one-time display, with a 2s "Copied" confirmation — removes the risk of mistyping while hand-copying a dense code under pressure. Both Android-only, both opt-in only in the sense that they're tied to App Lock already being on. `npx cap sync android` run for both |
| **5.1.0** | **Biometric unlock (B54, second half)** — fingerprint/face unlock, sitting alongside the PIN rather than replacing it (owner scope, 2026-07-20): a new opt-in toggle in Settings → App Lock (Android only, appears once a PIN exists), off by default. Uses `capacitor-native-biometric`, the closest thing to a de facto standard among community Capacitor biometric plugins since there's no official Ionic one. Enabling it requires a real successful fingerprint/face prompt, not just flipping a switch — a device with nothing enrolled gets a plain inline message instead of silently ending up "enabled" with no way to unlock. On the lock screen, the OS prompt now fires automatically the moment it opens (no extra tap), with the PIN input staying visible and usable throughout — a cancelled or failed biometric attempt falls straight through to the PIN screen rather than shaking like a wrong PIN would. The PIN itself is untouched: it's still the only thing that can set/change/remove the lock or drive "Forgot PIN" recovery. See DECISIONS.md's "Biometric unlock sits alongside the PIN" entry for the full reasoning. Still needs a live on-device confirmation in Android Studio before considering the actual fingerprint/face prompt fully verified (logic-level: build/tsc/145 tests all green, `npx cap sync android` run) |
| **5.0.0** | **App Lock, NLP Quick Add, recurring tasks, Time Travel, one font** — the largest feature batch since the pre-release audit, bumped to 5.0 for the sheer accumulated scope rather than any single breaking change. **App Lock (B54)**: a real PIN gate on the whole app (`config.ts`/`AppLock.svelte`) — cold-launch lock plus an idle/backgrounded timeout (1/5/15/30 min, Settings → App Lock), UI-gated via `inert` (blocks Tab/screen-reader access, not just visually hidden) rather than data encryption (see DECISIONS.md for why). Went through two rounds of owner-driven correction after shipping: the original "Forgot PIN" confirm button turned out to be rendering invisibly beneath the lock screen (a z-index bug, `ConfirmDialog` at 701 vs. the lock screen's 10001) — fixed by building the confirmation inline instead of using the shared dialog — and then, once visible, was recognized as fundamentally the wrong design anyway: a two-click bypass needing zero PIN knowledge. Replaced with a genuine one-time recovery code (shown once at PIN setup, must be explicitly acknowledged as saved, salted-hash stored exactly like the PIN) — the real security-relevant fix this release. Also added an optional self-written PIN hint, and four consistency fixes from the same feedback round: Cancel/Remove PIN buttons now match Settings' shared button style, "Lock after" became a proper dropdown instead of a cramped segmented control, and the biometric note is now gated to Android only (`isNativePlatform()`, exported from config.ts) instead of showing on PC/web where it can never apply. **NLP Quick Add**: typing `"Fix the login bug tomorrow 3pm #urgent @Work !!p1"` into Quick Add now parses dates, times, priority, tags, and project inline — fully local (`nlpParse.ts`, no network call, consistent with the project's no-telemetry stance), with inline syntax help and explicit escape hatches for anyone who'd rather type a literal title containing `#`/`@`/`!`. Audited afterward for a real bug: an unmatched `@mention` (no matching project) was leaking into date parsing instead of being left alone. **Recurring tasks**: daily/weekly/monthly repeat on any task; completing a repeating task resets it in place on its next due date rather than spawning a duplicate card, so history/attachments/position stay intact. **Time Travel**: replaces the old Changelog view with a proper journal — grouped by day, readable descriptions instead of raw log dumps, with the same audit-and-fix treatment (delete/move entries had wrong priority in the sort, some entity types were missing readable labels, a scroll-position bug on load) plus later polish (24h/12h time format setting, row-height/wrapping fixes, project name always its own row so it never mid-sentence wraps). **One font, project-wide**: IBM Plex Mono removed entirely (was only ever used for metadata-label styling, read as visually inconsistent next to Hanken Grotesk everywhere else, per owner feedback) — `--mono` now points at the same Hanken Grotesk face rather than a second `@font-face`. **Eleventh maintenance pass** (delta since v4.28.0, clean — no fixes needed): confirmed no dangling references survived the ChangelogView→TimeTravelView swap, Tauri CSP is genuinely enabled (not just claimed), App Lock/recovery-code secrets are never logged or stored in plaintext anywhere, and the audited try/catch+showError() invariant holds across every new mutation call site |
| **4.30.0** | **Pre-public-release audit, phase 1 (web) — batches 1-10** (owner-directed, 2026-07-18: "we need to have fully working functional thing, not provide public with minor bugs"). A systematic pass, batch by batch, over every major surface — 25+ real, verified bugs found and fixed, each with build/tsc/test green before moving to the next batch. **Sync**: the conflict badge only updated after a sync settled, so a session starting offline/paused with pre-existing conflicts showed nothing until reconnecting — now scanned once at startup too. **Core CRUD**: five entity mutations had zero changelog coverage despite `CLAUDE.md`'s own audited invariant — `undoDelete`, `unarchiveTask`, `deleteProject`, all of Space CRUD, and custom-field CRUD; also fixed `ChangelogView` mislabeling every create/delete as "task" regardless of actual entity type. **Views**: extended the `{#key}`-remount fix (see 4.29.0) to Dashboard/Focus/Agenda, missed in the first pass; Focus's locked commitment list now excludes archived tasks like every other read path. **Card Detail**: the lazy-loaded history panel's import had no error handling. **Quick Add/Search**: same remount-guard gap as Changelog/Trash/Settings, extended here too. **Settings**: all four Organize managers (Spaces/Tags/Custom Fields/Archived Projects) had both the unguarded-import bug and the remount-guard gap; corrected a stale comment claiming no updater config exists when one (a placeholder) does. **Notifications**: `_firedIds` was keyed by task id alone and never cleared — once any reminder fired on a task, a *new* reminder set on that same task later in the session would silently never fire again. **Data/Backup**: `importJSON()` (Restore) — its own UI copy promises a colliding doc "merges instead of duplicating," but the code stripped `_rev` unconditionally, so PouchDB rejected every collision with a 409 counted as "ok," silently leaving the existing (possibly stale) local doc untouched; now fetches the real `_rev` first so a restore genuinely overwrites. **Cross-cutting**: `confirm.ts`'s promise-based `confirmAction()` could leak a promise forever — a fast double-click triggering it twice before the first resolved silently overwrote the pending request, leaving the first caller's `await` hanging indefinitely; now resolves any still-pending request `false` before starting a new one. Final sweep confirmed all 15 `closeOnBack()` consumers in the codebase are now covered by the remount-guard pattern, documented as mandatory in `CLAUDE.md` for any future overlay. Phase 2 (PC quick check) and Phase 3 (Android quick check) still to come |
| **4.29.0** | **CSP enabled + light-sidebar redesign + a real modal-stack root-cause fix** (owner-directed, 2026-07-18, first pass of a planned pre-public-release audit). **Tauri CSP**: `security.csp` was `null` since first scaffolded — enabled a real policy (`script-src 'self'`, `style-src 'self' 'unsafe-inline'` for per-item dynamic colors, `connect-src 'self' http://*:*` for LAN sync/pairing to runtime-discovered addresses/ports — the `:*` matters, a bare `http://*` only matches the default port and silently broke sync on first try). Moved `index.html`'s one inline theme-flash script to `public/theme-init.js` so `script-src` can stay strict. **Sidebar now follows the page theme**: was pinned dark regardless of light/dark mode; now theme-aware via `--sidebar-bg` (light: `#FBFBFC`), with a new fixed-dark `--statusbar-fill` token split out so Android's status-bar strip (paired with an unconditional light-icon style) doesn't regress. Added a subtle hover shadow to sidebar rows. Darkened `--faint` (`#6B7280`→`#5F6674`) — measured 4.16:1 against `--hover`, under WCAG AA's 4.5:1; now clears AA everywhere it's used. **Settings resize**: tabs now animate smoothly between sizes (a measured FLIP, since a plain CSS `transition: height` can't animate an auto-sized element) instead of snapping instantly, and tab content fades in sync with it instead of popping. **The real bug, found by reproducing live, not by reasoning alone**: rapid open/close of Changelog could leave it permanently stuck — Escape/scrim/back all silently no-op. Root cause (two layers, both fixed): (1) `history.back()` calls can coalesce into a single `popstate`, which `modalStack.ts`'s old one-pop-per-event logic couldn't handle — rewritten to close every layer skipped, matched by a unique id per push rather than by depth (a depth-based first attempt broke after a page reload, since real browser history survives a reload but the in-memory stack doesn't). (2) The actual root cause: Svelte can *revive* a closing component instead of destroying and recreating it, if reopened before its outro finishes — the revived instance's own `closeOnBack()` never re-runs, so its `requestClose` is a stale, already-spent closure. Fixed by forcing a real remount on every open (`{#key}` + a bumped session counter) for Changelog/Trash/Settings and — since this is the same root cause as CardDetail's animation reading as "cut short" — Kanban/List/search-opened cards too. Documented as a mandatory pattern in `modalStack.ts` and `CLAUDE.md` for any future `closeOnBack()` consumer. **Also found auditing Ctrl+K end-to-end** (owner request, unrelated to the above): "Open Changelog"/"Open Settings"/"Open Deleted" from the command palette have likely never worked since this app moved to Svelte 5 — `Sidebar.svelte`'s `openChangelog`/`openTrash`/`openSettings` were plain functions, and Svelte 5 (unlike 3/4) requires an explicit `export` for a function to be callable via `bind:this` from a parent; fixed by exporting them. A second, separate bug in the same audit: `GlobalSearch.svelte`'s command-palette actions all used `requestClose()`, including the four that open another overlay — raced the new overlay's own `history.pushState()` exactly like the module's own docs already warned about for search results (which correctly use `discardTop()`); fixed by marking those four `opensOverlay: true` in `commands.ts` and routing them the same way |
| **4.28.0** | **Widget correction + pairing feedback + seed hardening** (owner-directed, 2026-07-17, live-tested from the previous release). **Widget size reverted**: v4.27.0's fix for the picker's clipped preview grew `minWidth`/`minHeight` from 110×110dp/2×2 cells to 200×160dp/4×2 — that also grew the REAL on-device footprint to 3×3, which was wrong; the 2×2 placement was already correct, only the preview thumbnail was cosmetically off. Reverted to 110×110dp/2×2 and left the preview quirk as a known, accepted cosmetic limitation of `previewLayout` rendering at thumbnail scale rather than distorting real behavior again to chase it. **Widget icon contrast**: the quick-add "+" glyph and the Dashboard grid icon were hardcoded `#FFFFFF`, sitting on `colorWidgetSurface` — invisible against light mode's (now cream) button circles. Both now use `colorWidgetText`, which already has correct light/dark values; the Focus pill's white text stays hardcoded on purpose, since it sits on the always-dark/saturated accent color, not the surface tone. **Pairing "stuck" feedback**: neither side of the Track E pairing flow showed any success confirmation — the phone's modal just silently reverted to its "Find my computer" scan screen after a successful pair (read as stuck/broken), and the PC has no direct signal a pairing even happened. Phone now shows a "✓ Connected to '<name>'" state with a Done button instead of reverting; PC polls `getDeviceLastSeen()` every 3s while its Connect modal is open and shows the same success line once a name appears that wasn't there when the code was generated, stopping the poll on modal close. **Seed hardening**: `seedIfEmpty()`'s `SEEDED_KEY` localStorage flag used to skip the emptiness check entirely once set — if the DB were ever legitimately empty again while that flag survived (owner reported a PC first launch with no spaces, root cause unconfirmed), the app would stay silently empty forever with no obvious recovery. New PC installer built (`Offlog_4.28.0_x64-setup.exe`) and verified to launch and stay running; `npx cap sync android` run for the widget/pairing/seed fixes, APK build/install remains an owner step in Android Studio |
| **4.27.0** | **Android widget polish + a real modal-stack bug** (owner-directed, 2026-07-17). **Widget preview**: the widget picker's preview (`previewLayout`, API 31+) renders `widget_offlog.xml` at the declared `minWidth`/`minHeight` (110×110dp) — the actual wrap_content card needs roughly 195×150dp (padding + the Focus pill + the 52dp button row), so the preview clipped/overlapped ("damaged") even though a real home screen's grid cell is normally bigger than the bare declared minimum and hid the mismatch there. Resized to 200×160dp to comfortably fit the real content; also dropped `maxResizeWidth`/`maxResizeHeight`, which were dead config — both are inert when `resizeMode="none"`, which this widget has always used. **Widget color**: light mode's card background/button-circle color was pulled directly from app.css's exact `--surface`/`--bg` light tokens (#FFFFFF/#F1F2F6) — read as too stark white against a real home-screen wallpaper (owner feedback). Warmed to a cream tint (`#FBF8F2`/`#F1EDE3`) specific to the widget's own `colors.xml`; dark mode (`values-night/colors.xml`) untouched, already fine. **Quick Add "stops working after repeated use"**: root-caused to `modalStack.ts`'s `closeOnBack()` — the returned `requestClose` function had no guard against being called more than once for the same overlay. Every overlay can reach it from multiple paths (Escape, a scrim click, a Cancel/Save button — QuickAdd's `doAdd()` also calls it on success), and `history.back()` resolves asynchronously via `popstate`; a second call before that resolves (a fast double-tap, or a click landing on a still-fading scrim right after Enter/Save already triggered one) fires a second `history.back()` against a stack that only ever had one entry pushed for that layer, over-navigating into whatever was underneath it — consistent with "works, then randomly doesn't" after repeated fast use. Now idempotent per layer. `npx cap sync android` run to carry all of this into the Android project; the actual APK build/install is, as always, an owner step in Android Studio |

| **4.25.0** | **First real desktop dogfooding round** (owner-directed, 2026-07-15/16) — the PC app went from "builds and runs" to an actual daily-use install, and testing it for real surfaced a long list of genuine bugs. **Real icon + first stable release build**: `offlog-desktop`'s icons were still Tauri's scaffold placeholders — regenerated from the same brand-composited source Android's launcher uses, rounded corners baked in (Windows has no adaptive-icon masking like Android's), version bumped off the `0.1.0` placeholder. **Startup**: the visible console window during launch was `couchdb.cmd` (a batch file) spawning unsuppressed — `CREATE_NO_WINDOW` fixed it; closing that window used to kill sync independently of the app. The ~10s blank-window delay was `.setup()` blocking on the full CouchDB boot before Tauri would paint anything — moved to a background task, then went further: the window now starts hidden and only reveals itself once the frontend's first real paint has happened, so there's no blank-then-pop-in flash at all. **Reinstall corruption**: "no db shards could be opened" after an uninstall/reinstall cycle was CouchDB's actual data living inside the installed-program directory, which a reinstall can partially wipe while a fresh install regenerates a new node identity — moved the real database to `app_data_dir` (never touched by the installer), which also means data now survives a clean reinstall instead of vanishing. **Notifications**: permission could finally be requested (desktop was silently using the plain browser Notification API, which Tauri's WebView2 has no prompt handler for) via `tauri-plugin-notification`, but two more real gaps followed — its desktop backend never reads the `schedule` field at all (scheduling is mobile-only in that plugin), so reminders never fired until rewritten onto a JS timer that displays a real toast at the right moment; Also found: `isPermissionGranted()`/`requestPermission()` are hardcoded to always return granted on desktop — there's no real permission model behind them, which explains why the UI seemed to randomly re-block itself. **Backup/Export**: same blob-URL-has-no-download-manager gap A34 already found on Android's WebView, just never extended to Tauri's WebView2 — `tauri-plugin-fs`/`tauri-plugin-dialog` now give it a real native "Save As" dialog with a pre-filled filename and extension filter. **Security audit** (full pass, see git log for the complete report): one real fix, a brute-force lockout on the LAN pairing code (was unlimited guesses within its 5-minute window, now locked out after 8 wrong attempts); everything else (XSS surface, CORS config, credential logging, Tauri command surface, dependencies) reviewed clean. **Also shipped**: per-space icon picker (25 icons — new spaces used to silently reuse a generic icon that read as "another Dashboard link"), Dashboard's Today/Overdue lists no longer cram title+project+date onto one truncated line, Agenda's week view no longer stretches empty day columns to fill the whole viewport, `TimePicker.svelte` now offers every minute instead of 5-minute steps, week now defaults to starting Monday, and the bundled CouchDB dropped ~64MB of genuinely unused Fauxton/Nouveau files. **Scaffolded, not functional yet**: a desktop auto-updater (`tauri-plugin-updater`) — registered and wired into Settings, deliberately unconfigured until C1 provides real hosting and the owner generates a real signing key (see ROADMAP.md's new E3 entry) |
