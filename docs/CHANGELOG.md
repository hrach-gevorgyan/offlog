# Offlog — Changelog

The single source of truth for version history — this used to be duplicated
across `TECH.md` (technical detail) and the root `README.md` (short
highlights), which had drifted in coverage (TECH.md's table was more
complete). Consolidated here 2026-07 as part of the documentation cleanup;
both `README.md` and `docs/TECH.md` now link here instead of keeping their
own copy. New entries go at the top, one row per released version.

This table keeps the newest **10** releases in full detail. Older releases
are compressed to one line each in
[archive/changelog-archive.md](archive/changelog-archive.md) (git still has
every full diff; that folder also holds the full maintenance-pass log,
moved out of MAINTENANCE.md 2026-07-20) — split out 2026-07 once this
file's default-loaded size grew large enough to matter. When adding a new
entry, move the oldest row here into the archive (compressed to one line)
once this table exceeds 10 rows, so it stays roughly the same length
release over release. Archive weekly, or whenever it's due, not on a fixed
per-release schedule.

| **5.7.1** | **CRITICAL — real dev credentials were baked into production builds** (owner live-testing on Android, 2026-07-21: onboarding never offered sync setup on a fresh install, and the sidebar showed a 4th sync button with an error state despite never pairing). Root cause: Vite loads `.env.local` for every build mode, not just `npm run dev` — the developer's own `.env.local` (real self-hosted CouchDB URL + username + password, meant purely for local dev convenience) was compiled directly into `dist/`, and from there into the just-published v5.7.0 GitHub Release's Android APK, in plaintext. Confirmed: the real password was recoverable straight out of the shipped APK. Fixed by gating `config.ts`'s `envUrl`/`envUser`/`envPass` reads on `import.meta.env.DEV`, which Vite resolves statically at build time — a production build now has these as literal `undefined`, so the minifier removes the actual secret values from the bundle entirely rather than just skipping their use at runtime. Verified: rebuilt `dist/` no longer contains the credential anywhere. The v5.7.0 Release (and its leaked APK) was deleted; the real CouchDB password needs rotating separately since it left git's reach entirely (distributed in a binary file, not through the repository — `dist/` itself was never committed). Process gap closed: `docs/MAINTENANCE.md`'s security checklist now explicitly requires checking the actual built `dist/` output for `.env.local` leakage every pass, not just source files and git history — a source-only secret scan structurally cannot catch this class of bug. **Also from the same live-testing session** (this version wasn't released yet, so folded in rather than a separate bump): widget taps not switching views on a warm start on Samsung/OneUI (`FLAG_ACTIVITY_SINGLE_TOP` missing from the widget's `PendingIntent`); today/due-date math used UTC instead of local calendar date in 7 places (Agenda's "This week" view, `utils.ts`'s shared `TODAY()`/`dueLabel`/`dueRelative`, Focus's daily commitment lock, Global Search, Kanban's overdue badge, 2 export filenames) — consolidated to one shared `localDateStr()` (moved from `db.ts` to `utils.ts` to avoid a circular import); Kanban board didn't auto-scroll toward an off-screen column while dragging a card near the screen edge. **A third round, same still-unreleased version**: `isSyncEnabled()` defaulted true unconditionally, so a native device that skipped the onboarding sync offer still showed a misleading "Sync enabled" toggle in Settings — now defaults true only when a real `syncUrl` actually exists. Sidebar's bottom-row footer was a fixed 2-column grid (built for exactly 4 buttons) that left the 3rd button wrapping alone when sync's button is hidden — new 3-equal-column layout for that case. Sync tab and Advanced tab's manual-connection section now collapse to just the enable toggle when sync is off (same pattern App Lock uses), and the footer button only says "Save & restart sync" when sync is actually on. Time-format toggle relabeled from abstract "24h"/"12h" to concrete example times ("1:00 PM"/"13:00"); Privacy Screen's description rewritten to lead with what it actually does. Also this round: removed the unused `@capacitor/assets` devDependency (cleared 15 of 17 Dependabot advisories — it was pulling in the entire node-tar/xcode/minimatch chain), an unused `--dur-fast` CSS token, 2 orphaned icon assets, and a stale code comment |

| **5.6.2** | **Archived-project tasks leaking into every cross-project view** (owner-reported, 2026-07-21: done tasks from an archived project still showing on Dashboard/Agenda/Search, project name rendering as "—"). Root cause: `archiveProject()` only sweeps a project's *non-done* tasks into `archived:true` (done tasks are deliberately left alone, by design — see its own comment), so a task already sitting in the last column when its project got archived kept `archived:false` forever. Every cross-project query filtered tasks solely on that flag, never on whether the task's own project was archived — six leaked it: `getDashboardData()` (pinned/overdue/today, and the source of the "—" placeholder), `searchAllTasks()` (Global Search), `getOpenTasksForFocusPicker()` (a leaked task was even pickable in Focus), `getAllActiveTasksWithReminders()` (could still fire a real reminder notification for a task in an archived project), `getAllTasksDue()` (Agenda), and `getRecentlyModifiedTasks()` (Sidebar's Recent list). All six now also require the task's project to still be active. Confirmed unaffected and correctly left alone: `ArchivedProjectsManager.svelte` (Settings — the intended one place archived-project detail should live), the per-project "Show archived" toggle in List view, Trash, and CSV/JSON export. Verified live in the browser (not just by code trace): archived a project with a pinned, overdue, done task and confirmed it fully disappeared from all four surfaces with no "—" left behind. `db.ts`-only change, no schema/sync/test changes needed |

| **5.6.0** | **Duplicate-name/content nudges** (owner-reported, 2026-07-20: spotted a real duplicate "Draft" project after two independently-seeded devices merged). Non-blocking, never-blocking hints — same-name projects/spaces/tasks and similar notes are all legitimate, this only helps notice when they weren't intentional. | **5.5.1** | **S2 live-verified + a real conflict bug fixed** — paired a real `offlog-desktop` host against a realistic 180-doc dataset (112 tasks, 13 projects, 4 spaces, one default space at revision 15) to verify the mobile-then-PC merge scenario from docs/IDEAS.md's S2. Replication itself is sound: zero data loss, every doc byte-for-byte identical on both sides. Found a real gap in the process: `clearLocalSeedBeforeFirstPair()` only clears a device's own copy of the 4 fixed default-seed ids when that device has zero tasks, so a phone with real accumulated history (this scenario) skips the guard — its independently-created copies of `space:unsorted`/`personal`/`work`/`project:draft` genuinely conflicted with the host's own freshly-seeded copies (3 real conflicts, confirmed via `scanConflicts()`). Not data loss in this run (CouchDB's deterministic winner happened to favor the real content each time) but not guaranteed. Fixed at the root: `scanConflicts()` now auto-resolves a conflict on one of those 4 known ids whenever one side is provably still the untouched pristine default — discards a pristine loser, or adopts a real edit that lost to a still-pristine "winner." Two genuinely different real edits are still left as a real conflict for manual resolution, same as always. 2 new tests in `tests/db.test.ts` |
| **5.5.0** | **Sync-architecture hardening, first pass** — two additive fixes from docs/IDEAS.md's sync-topology scenarios (owner review, 2026-07-20), neither touching the actual replication path (`db.ts`'s `sync()` call). **Multiple-host detection**: `offlog-desktop` previously spawned its own CouchDB sidecar with zero awareness of another Offlog host already on the LAN — two PCs silently became two independent islands with no warning. New `discovery.rs::browse_for_others()` runs a one-time mDNS scan at startup (before this instance advertises itself, so it only sees genuinely other hosts), exposed via a new `get_detected_other_hosts` Tauri command; Settings → Sync now shows a warning if one is found. Deliberately *not* a full "join as client instead" mode — that's a much bigger feature for a scenario nobody's hit yet. **Stale-host silent failure, a real bug fix**: `discovery.ts`'s `findPairedHostAddress()` used to silently ignore any mDNS advertisement whose uuid didn't match the paired host's, meaning a genuinely-changed host identity (wiped/reinstalled PC, wrong-device pairing) just failed forever with zero user-facing signal beyond a generic "cannot reach sync server." Now surfaces a `staleHostAlert` the moment it sees a *different* host but can't find the paired one — shown as an actionable "re-pair?" tooltip/badge on the Sidebar sync button. Also resolved by code-read: sync-host identity and CouchDB data already share `app_data_dir` and both survive a normal uninstall/reinstall (an earlier fix), so host-loss risk is narrower than first assumed — only a full OS-profile wipe orphans paired phones. Full scenario-by-scenario status in docs/IDEAS.md |
| **5.4.6** | **Maintenance pass (thirteenth run)**, delta-scoped since v5.4.0 (covering v5.4.1-v5.4.5's rapid live-device bugfix batches). Found and fixed one real race: `notifications.ts`'s `fireWebNotification()` cleared a fired reminder's `reminder_at` via an un-awaited `updateTask(...).catch(()=>{})`; since `updateTask` is a plain get-then-put (no compare-and-swap), a second `updateTask` on the same task starting before that fire-and-forget write finished could read a stale rev and throw "Document update conflict" — reproduced intermittently in `tests/notifications.test.ts`. Fixed generally in `db.ts` by serializing all `updateTask()` calls per task id through a small write queue, rather than patching just this one call site. Also extracted `KanbanBoard.svelte`'s repeated touch-drag-state reset (`onTouchEnd`/`onTouchCancel`) into one `resetTouchDragState()` helper, and added an explicit release-checklist note in CLAUDE.md that the Android `release` build type is currently signed with AGP's public debug keystore (v5.4.4, local-dev convenience only) and must not go out as a real Play Store build before C3's real signing key exists. No dead code, no dependency changes, `npm audit`/security checklist unchanged from prior passes |
| **5.4.5** | **Widget clicks getting stuck on the first-opened view** (owner-reported, 2026-07-21: after Quick Add opened once, every later widget tap re-opened Focus regardless of which button was pressed). Root cause: `closeAll()` (added in 5.4.3) only issued `history.go(-n)` and relied on the *async* `popstate` it eventually produces to actually pop the modal stack and close things — rapid widget taps could get that navigation coalesced away by the browser (the exact same failure class `modalStack.ts`'s own comments already document for `history.back()`), leaving the stack stale so the next tap read the wrong state. Rewritten to be fully synchronous: pops and closes every entry immediately, with the `history.go()` now purely a best-effort compensating step for a later hardware back-press. `handleWidgetUrl()` also switched from `url.includes(host)` substring matching to exact `new URL(url).hostname` parsing — not the actual bug, but more correct and matches how the `project` URL already had to be parsed. **Widget preview now realistic-but-small**: the dedicated preview layout (5.4.3) is `wrap_content` inside a fixed frame, so the earlier 200×170dp size mostly added empty space *around* the card once the picker scaled the whole frame down to thumbnail size — tightened to 185×155dp, close to the card's natural size, so it fills most of the frame instead of floating in the middle of it |
| **5.4.4** | **Fixed Android Studio's "Activity class ... does not exist" running `release`** (owner-reported, 2026-07-21). Not a real missing-activity problem — the `release` build type had no `signingConfig` at all, so it was unsigned and Studio's Run button couldn't actually install it. Pointed it at AGP's built-in debug keystore, the standard local-dev pattern for running the release build type directly; a real Play Store release still needs its own signing key when that's actually being prepared (see ROADMAP.md's C1). `debug`'s `applicationIdSuffix` (a deliberate earlier fix for storage collisions between debug/real installs) is untouched — this only makes `release` runnable, doesn't remove the debug variant |
