# Offlog — Changelog

The single source of truth for version history — this used to be duplicated
across `TECH.md` (technical detail) and the root `README.md` (short
highlights), which had drifted in coverage (TECH.md's table was more
complete). Consolidated here 2026-07 as part of the documentation cleanup;
both `README.md` and `docs/TECH.md` now link here instead of keeping their
own copy. New entries go at the top, one row per released version.

This table keeps the newest **10** releases in full detail. Older releases
are compressed to one line each in
[archive/changelog-archive.md](archive/changelog-archive.md) (git still has
every full diff; that folder also holds the full maintenance-pass log,
moved out of MAINTENANCE.md 2026-07-20) — split out 2026-07 once this
file's default-loaded size grew large enough to matter. When adding a new
entry, move the oldest row here into the archive (compressed to one line)
once this table exceeds 10 rows, so it stays roughly the same length
release over release. Archive weekly, or whenever it's due, not on a fixed
per-release schedule.

| Version | Changes |
|---|---|
| **5.4.6** | **Maintenance pass (thirteenth run)**, delta-scoped since v5.4.0 (covering v5.4.1-v5.4.5's rapid live-device bugfix batches). Found and fixed one real race: `notifications.ts`'s `fireWebNotification()` cleared a fired reminder's `reminder_at` via an un-awaited `updateTask(...).catch(()=>{})`; since `updateTask` is a plain get-then-put (no compare-and-swap), a second `updateTask` on the same task starting before that fire-and-forget write finished could read a stale rev and throw "Document update conflict" — reproduced intermittently in `tests/notifications.test.ts`. Fixed generally in `db.ts` by serializing all `updateTask()` calls per task id through a small write queue, rather than patching just this one call site. Also extracted `KanbanBoard.svelte`'s repeated touch-drag-state reset (`onTouchEnd`/`onTouchCancel`) into one `resetTouchDragState()` helper, and added an explicit release-checklist note in CLAUDE.md that the Android `release` build type is currently signed with AGP's public debug keystore (v5.4.4, local-dev convenience only) and must not go out as a real Play Store build before C3's real signing key exists. No dead code, no dependency changes, `npm audit`/security checklist unchanged from prior passes |
| **5.4.5** | **Widget clicks getting stuck on the first-opened view** (owner-reported, 2026-07-21: after Quick Add opened once, every later widget tap re-opened Focus regardless of which button was pressed). Root cause: `closeAll()` (added in 5.4.3) only issued `history.go(-n)` and relied on the *async* `popstate` it eventually produces to actually pop the modal stack and close things — rapid widget taps could get that navigation coalesced away by the browser (the exact same failure class `modalStack.ts`'s own comments already document for `history.back()`), leaving the stack stale so the next tap read the wrong state. Rewritten to be fully synchronous: pops and closes every entry immediately, with the `history.go()` now purely a best-effort compensating step for a later hardware back-press. `handleWidgetUrl()` also switched from `url.includes(host)` substring matching to exact `new URL(url).hostname` parsing — not the actual bug, but more correct and matches how the `project` URL already had to be parsed. **Widget preview now realistic-but-small**: the dedicated preview layout (5.4.3) is `wrap_content` inside a fixed frame, so the earlier 200×170dp size mostly added empty space *around* the card once the picker scaled the whole frame down to thumbnail size — tightened to 185×155dp, close to the card's natural size, so it fills most of the frame instead of floating in the middle of it |
| **5.4.4** | **Fixed Android Studio's "Activity class ... does not exist" running `release`** (owner-reported, 2026-07-21). Not a real missing-activity problem — the `release` build type had no `signingConfig` at all, so it was unsigned and Studio's Run button couldn't actually install it. Pointed it at AGP's built-in debug keystore, the standard local-dev pattern for running the release build type directly; a real Play Store release still needs its own signing key when that's actually being prepared (see ROADMAP.md's C1). `debug`'s `applicationIdSuffix` (a deliberate earlier fix for storage collisions between debug/real installs) is untouched — this only makes `release` runnable, doesn't remove the debug variant |
| **5.4.3** | **5 more real bugs from a second live-device pass, one with a screenshot** (owner-reported, 2026-07-21). **Widget picker preview showed "FOCUS" wrapped one letter per line**: `previewLayout` reused the real `@layout/widget_offlog`, whose root is `match_parent` — the picker measures that against `minWidth`/`minHeight` (110×110dp, correct for the real 2×2 footprint, must never change again after the 4.27.0/4.28.0 revert), squishing the wrap_content card into a near-square canvas the Focus pill's text can't fit in. New dedicated `@layout/widget_offlog_preview` with its own fixed 200×170dp root — fully separate from `initialLayout`, zero effect on the real on-device size. **Widget taps overlapping instead of replacing** (Quick Add left open behind Dashboard, same for Focus): `handleWidgetUrl()` only flipped view-level booleans, never closed whatever overlay was already open on a warm start. New `modalStack.ts` export `closeAll()` — one real `history.go()` back past every pushed layer, reusing `onPopState`'s existing "id absent → close everything" fallback instead of a new bypass path — called at the top of every widget-url handle. **Drag-and-drop ghost card sometimes stayed on screen** after a long-press, even without completing a drop, persisting across navigation: only `touchend` was wired, not `touchcancel` (which Android fires instead when a system gesture, notification-shade pull, or backgrounding interrupts the sequence) — added a cancel handler, plus an `onDestroy` safety net since the ghost element is appended straight to `document.body`, outside Svelte's own tree, so navigating away from Kanban was never enough to clean it up on its own. **Privacy Screen silently blocked all screenshots**, not just the recents-switcher preview it was meant to hide — that's what Android's FLAG_SECURE (what `PrivacyScreen.enable()` actually sets) does, and there's no way to have one without the other. It shouldn't have been bundled automatically into turning on a PIN; now a separate, explicit, **off-by-default** toggle (Settings → App Lock → Privacy screen) |
| **5.4.2** | **7 real bugs from a full live-device pass** (owner-reported, 2026-07-21). **Overlays sometimes not closing, needing a force-restart** (CardDetail Save, Time Travel/other X buttons, Settings Cancel): `modalStack.ts`'s `requestClose()` calls `history.back()` and waits for the resulting `popstate` to actually pop and close the layer — but that's a request, not a guarantee, and Android WebView has been seen to occasionally not deliver it at all; since the guard against double-firing is single-shot, a dropped `popstate` meant that overlay could never be retried. Added a 400ms fallback: if the entry hasn't been removed by then, close it directly instead of waiting forever on an event that may never arrive — a no-op in the normal case, since by then `popstate` has almost always already handled it. **Settings' Save button opened the PIN lock screen even with nothing changed**: `saveSettings()` called `location.reload()` unconditionally on every tab, not just Sync/Advanced — every other tab's settings already apply live on interaction, only the sync URL/credentials are buffered waiting for Save, so a reload on any other tab was pure side effect, and with App Lock on, a reload re-triggers the cold-start lock check. Now only reloads if the sync URL/credentials actually changed. This also removes a real contributor to "locks by itself constantly." **Quick Add's floating button stayed visible and clickable on top of Settings**: it only hid for `CardDetail`'s `modalOpen` store; Settings never set it. Now does, on every platform. **Focus's 3 committed tasks never visually reflected "done"**: `markDone()` genuinely updated the task (confirmed via Time Travel) but the row template had no done-state check at all — same-looking row forever, on every platform. Added the checked-circle/strikethrough treatment List already has. **PIN input polish**: the lock screen's centered PIN field left a distracting blinking caret jumping to the middle of the box as you type — hidden via `caret-color: transparent` (standard fix for centered PIN/OTP-style fields); Settings' 4 PIN setup/change inputs were missing `autocomplete="off"` (present on the lock screen's own input but not these), which can trigger Android's password-autofill overlay. **Splash logo was shrunk**: `windowSplashScreenAnimatedIcon` (the real API 31+ splash icon, used on nearly every device) pointed at the launcher's adaptive-icon foreground, which already has the 66%-of-canvas safe-zone padding launcher icons need — the SplashScreen API applies its *own* additional inset on top, stacking two rounds of padding. New dedicated `drawable/splash_icon.png` (mark at 92% of canvas, transparent background) fixes it. **Widget light mode still looked flat ("dark is good")**: not an icon-contrast issue (already fixed, v4.28.0) — `colorWidgetSurface` (the button circles) was only ~10/255 per channel off `colorWidgetBg` (the card), so the buttons barely stood out from the card itself in light mode; dark mode's equivalent gap is 2-3x bigger in relative terms. Deepened the light surface tone to a real, visible step while staying in the warm-cream family. Widget preview-thumbnail mismatch and inconsistent widget-tap PIN prompting are known, already-explained tradeoffs (Android's picker renders at declared minWidth/minHeight, smaller than the real card; the PIN prompt inconsistency is expected Android process-lifecycle behavior — a cold-killed process re-locks, a warm-resumed one doesn't within the idle timeout) — not re-touched blind |
| **5.4.1** | **Maintenance pass (twelfth run)**, delta-scoped since v5.0.0 — covering biometric unlock, Privacy Screen, Clipboard, Haptics, App Launcher, and the Android cleanup. Found and fixed one real inconsistency: `hapticToggle()` fired *before* the task mutation was confirmed in 4 places (List/Focus/Deadlines' `markDone`, Kanban's `togglePin`) — a buzz would fire even if the save then failed, unlike the drag-drop haptic calls (already correct, firing only after `updateTask` succeeds). All 4 moved after the `await`. Everything else audited clean: all 5 new plugins from this arc genuinely used, no plaintext secrets in the 2 new localStorage flags, `clearAppLockPin()` correctly clears the biometric flag too, `npm audit` unchanged from prior passes. `SettingsPanel.svelte` flagged again at 1881 lines (was 1141 three passes ago) — same shared-CSS blocker, still deliberately deferred |
| **5.4.0** | **App Launcher (B57)** — when App Lock's biometric toggle finds nothing enrolled, an "Open enrollment settings" button now jumps straight to Android's biometric enrollment screen (`android.settings.BIOMETRIC_ENROLL`, API 30+) instead of just telling the user to go find it themselves, falling back to the generic Security settings screen on older devices (API 24-29, this app's minSdk floor, where `BIOMETRIC_ENROLL` doesn't exist). Only shown for the specific "nothing enrolled" failure, not any other biometric error a settings trip wouldn't fix. Technical note: `@capacitor/app-launcher`'s `openUrl()` isn't documented to accept raw Settings intent-action strings, but its Android implementation falls through to `new Intent(url)` when the input isn't a URL or package name — confirmed by reading the plugin's own Android source rather than assumed. Still needs a live on-device check to confirm the right screen actually opens across real OEM skins |
| **5.3.0** | **Haptics (B58)** — tactile feedback on checkbox/pin/checklist toggles and Kanban drag-and-drop, via a new shared `src/lib/haptics.ts` (centralizes the platform/setting gate in one place instead of re-checking at every call site). Wired into List/Focus/Deadlines' mark-done, CardDetail's checklist toggle, Kanban's pin toggle, and both Kanban drag paths — the HTML5 desktop one and, more importantly, the touch-based mobile one that's what actually runs on Android. No swipe actions exist in this codebase, so that part of the original scope didn't apply. New Settings → View & Accessibility toggle, Android only, **defaults on** — unlike biometric's security-driven opt-in default, there's no tradeoff to a vibration defaulting on, so it ships the way most native apps do |
