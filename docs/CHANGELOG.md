# Offlog — Changelog

The single source of truth for version history — this used to be duplicated
across `TECH.md` (technical detail) and the root `README.md` (short
highlights), which had drifted in coverage (TECH.md's table was more
complete). Consolidated here 2026-07 as part of the documentation cleanup;
both `README.md` and `docs/TECH.md` now link here instead of keeping their
own copy. New entries go at the top, one row per released version.

This table keeps the newest ~8 releases in full detail. Older releases are
compressed to one line each in [CHANGELOG-ARCHIVE.md](CHANGELOG-ARCHIVE.md)
(git still has every full diff) — split out 2026-07 once this file's
default-loaded size grew large enough to matter. When adding a new entry,
move the oldest row here into the archive (compressed to one line) so this
table stays roughly the same length release over release.

| **4.27.0** | **Android widget polish + a real modal-stack bug** (owner-directed, 2026-07-17). **Widget preview**: the widget picker's preview (`previewLayout`, API 31+) renders `widget_offlog.xml` at the declared `minWidth`/`minHeight` (110×110dp) — the actual wrap_content card needs roughly 195×150dp (padding + the Focus pill + the 52dp button row), so the preview clipped/overlapped ("damaged") even though a real home screen's grid cell is normally bigger than the bare declared minimum and hid the mismatch there. Resized to 200×160dp to comfortably fit the real content; also dropped `maxResizeWidth`/`maxResizeHeight`, which were dead config — both are inert when `resizeMode="none"`, which this widget has always used. **Widget color**: light mode's card background/button-circle color was pulled directly from app.css's exact `--surface`/`--bg` light tokens (#FFFFFF/#F1F2F6) — read as too stark white against a real home-screen wallpaper (owner feedback). Warmed to a cream tint (`#FBF8F2`/`#F1EDE3`) specific to the widget's own `colors.xml`; dark mode (`values-night/colors.xml`) untouched, already fine. **Quick Add "stops working after repeated use"**: root-caused to `modalStack.ts`'s `closeOnBack()` — the returned `requestClose` function had no guard against being called more than once for the same overlay. Every overlay can reach it from multiple paths (Escape, a scrim click, a Cancel/Save button — QuickAdd's `doAdd()` also calls it on success), and `history.back()` resolves asynchronously via `popstate`; a second call before that resolves (a fast double-tap, or a click landing on a still-fading scrim right after Enter/Save already triggered one) fires a second `history.back()` against a stack that only ever had one entry pushed for that layer, over-navigating into whatever was underneath it — consistent with "works, then randomly doesn't" after repeated fast use. Now idempotent per layer. `npx cap sync android` run to carry all of this into the Android project; the actual APK build/install is, as always, an owner step in Android Studio |
| **4.26.0** | **Settings redesign + mobile-info-loss fixes** (owner-directed, 2026-07-16). **Settings**: full rewrite of `SettingsPanel.svelte` (was a genuine "different designer per tab" mess by the end of the pass) into one consistent visual language — every tab is now a stack of titled `.setting-group` cards (accent-dot title, then rows), no bare rows on the panel background, no section dividers (card boundaries do that job). Multi-step flows (device pairing, conflict resolution, the maintenance run, restore-from-backup preview) moved into small in-file modals instead of living permanently in a tab, each with a proper right-aligned actions footer and a shared `.btn-primary` accent style — all built inline in `SettingsPanel.svelte`, no new component files. Maintenance folded into a new Advanced tab (the one deliberately technical tab, holding the raw CouchDB connection fields too) instead of being its own always-visible category. New Reduce Motion toggle (`theme.ts`'s `prefersReducedMotion()`, read live by `motion.ts` via getters so no call site elsewhere needed to change) sits under a real Accessibility group alongside High Contrast. Fixed along the way: the modal wasn't actually centering (`.mini-modal` was missing the static `transform: translate(-50%,-50%)` that `dialogPop`'s transition only applies transiently); dark mode's cards read as heavy dark blocks because `--bg` is *darker* than the panel's own `--surface` in dark theme — switched to a subtle `color-mix` tint instead of a flat `--bg` fill; the panel's fixed `height` (meant to stop per-tab resizing) left a wall of empty space on short tabs/mobile's category list — back to `max-height` so it fits content, capped and scrollable only when content actually overflows. **Notification click-to-open** now also calls the new `show_main_window` Tauri command so the desktop window actually comes forward, not just the card. **Mobile info loss** (same root bug, found in three places once looked for): `DashboardView.svelte`'s `.task-proj { display: none }` at ≤600px hid project name *and* the Overdue day-count with it — stale leftover from before the two-line row redesign, removed (Pinned's row also got the two-line treatment it had been missed for). `DeadlinesView.svelte` and `FocusView.svelte` had the identical pattern (`.proj-badge { display: none }` on their Agenda/Focus task rows) — same fix, title+project now stack on their own line instead of the project vanishing below ~700px. Also fixed: the shared mobile header pattern (Dashboard/Agenda/Focus, each a near-duplicate `<hamburger><title-block>` row) used `align-items: center`, which put the hamburger at a different height depending on how many subtitle lines that page has — `align-items: flex-start` makes it consistently align with the title's first line everywhere, with `align-self: center` on the header's own right-side controls (Agenda's List/Week toggle, Focus's Reset button) so those stay centered independent of the title block's height |
| **4.25.0** | **First real desktop dogfooding round** (owner-directed, 2026-07-15/16) — the PC app went from "builds and runs" to an actual daily-use install, and testing it for real surfaced a long list of genuine bugs. **Real icon + first stable release build**: `offlog-desktop`'s icons were still Tauri's scaffold placeholders — regenerated from the same brand-composited source Android's launcher uses, rounded corners baked in (Windows has no adaptive-icon masking like Android's), version bumped off the `0.1.0` placeholder. **Startup**: the visible console window during launch was `couchdb.cmd` (a batch file) spawning unsuppressed — `CREATE_NO_WINDOW` fixed it; closing that window used to kill sync independently of the app. The ~10s blank-window delay was `.setup()` blocking on the full CouchDB boot before Tauri would paint anything — moved to a background task, then went further: the window now starts hidden and only reveals itself once the frontend's first real paint has happened, so there's no blank-then-pop-in flash at all. **Reinstall corruption**: "no db shards could be opened" after an uninstall/reinstall cycle was CouchDB's actual data living inside the installed-program directory, which a reinstall can partially wipe while a fresh install regenerates a new node identity — moved the real database to `app_data_dir` (never touched by the installer), which also means data now survives a clean reinstall instead of vanishing. **Notifications**: permission could finally be requested (desktop was silently using the plain browser Notification API, which Tauri's WebView2 has no prompt handler for) via `tauri-plugin-notification`, but two more real gaps followed — its desktop backend never reads the `schedule` field at all (scheduling is mobile-only in that plugin), so reminders never fired until rewritten onto a JS timer that displays a real toast at the right moment; and it never wires up any click/action callback either, so tapping a notification did nothing until a custom Rust command (`send_task_notification`) started building toasts directly with `tauri-winrt-notification`, whose `on_activated` callback actually works. Also found: `isPermissionGranted()`/`requestPermission()` are hardcoded to always return granted on desktop — there's no real permission model behind them, which explains why the UI seemed to randomly re-block itself. **Backup/Export**: same blob-URL-has-no-download-manager gap A34 already found on Android's WebView, just never extended to Tauri's WebView2 — `tauri-plugin-fs`/`tauri-plugin-dialog` now give it a real native "Save As" dialog with a pre-filled filename and extension filter. **Security audit** (full pass, see git log for the complete report): one real fix, a brute-force lockout on the LAN pairing code (was unlimited guesses within its 5-minute window, now locked out after 8 wrong attempts); everything else (XSS surface, CORS config, credential logging, Tauri command surface, dependencies) reviewed clean. **Also shipped**: per-space icon picker (25 icons — new spaces used to silently reuse a generic icon that read as "another Dashboard link"), Dashboard's Today/Overdue lists no longer cram title+project+date onto one truncated line, Agenda's week view no longer stretches empty day columns to fill the whole viewport, `TimePicker.svelte` now offers every minute instead of 5-minute steps, week now defaults to starting Monday, and the bundled CouchDB dropped ~64MB of genuinely unused Fauxton/Nouveau files. **Scaffolded, not functional yet**: a desktop auto-updater (`tauri-plugin-updater`) — registered and wired into Settings, deliberately unconfigured until C1 provides real hosting and the owner generates a real signing key (see ROADMAP.md's new E3 entry) |
| **4.24.0** | **C2/C10 — zero-config first-run + plain-language pass** (owner-directed, 2026-07-15). **C2**: audited every screen a brand-new install (0 spaces/projects/tasks) would actually see. Found two real gaps — `DashboardView.svelte`'s project grid had no empty-state message at all with 0 projects (rendered as a bare blank area), and `KanbanBoard.svelte` had no empty-state hint for a project with 0 tasks either. Both fixed with a plain one-line hint pointing at the actual next action ("+ New project" in the sidebar / "+ Add card" in any column). Verified live against a genuine fresh install (cleared PouchDB + localStorage, not just PouchDB — the `SEEDED_KEY` flag gates the real auto-seed of 3 starter spaces, so a partial wipe doesn't reproduce a true first run). **C10**: reworded the Restore section's `Import JSON` button to `Choose backup file` and its jargon "A doc whose id already exists merges instead of duplicating" line to plain language (both were outside the Developer options section, so not deliberately technical); softened the app's crash-recovery screen (`App.svelte`) — the raw JS error message is now clearly labeled "Details:" below a plainer explanation instead of leading with it, and the backup suggestion no longer assumes Settings is reachable mid-crash; fixed an awkward "{name}"'s screen" pairing-code phrasing; softened desktop-web's "see Developer options below to connect" sync-status wording. README.md and the rest of the Developer-options block were checked and left as-is — that's the correct audience-appropriate register for each (developer README, deliberately technical settings) |
| **4.23.0** | **B39, B48, B50 — device identity + custom time picker + widget polish** (owner-directed, 2026-07-15). **B39**: renaming a device used to leave its old name stuck in "Devices seen recently" and card history forever, since the free-text name was the *only* thing identifying a device — every past `log:` doc kept whatever name was current when it was written. Added a stable per-install id (`getDeviceId()`, `config.ts`, a persisted UUID that never changes) stamped as a new `source_id` field on every new log doc alongside the existing `source` display name; `getDeviceLastSeen()` now groups by `source_id` where present, falling back to the literal `source` string for log entries written before this field existed — additive-only, no migration needed. **B50**: the themed `CalendarPicker.svelte` (B38) already replaced the native date input, but its time-of-day row and the Notifications default-reminder-time field both still fell back to a native `<input type="time">`. New `TimePicker.svelte` — two `CustomSelect.svelte` dropdowns (hour, 5-minute increments) sharing the app's existing picker look — replaces both. **B48**: Android widget was hardcoded to always render dark (no `values-night/` directory existed at all), regardless of the OS's own light/dark setting — split `colorWidget*` into a light default (`values/colors.xml`) and a dark override (`values-night/colors.xml`, holding the previous hardcoded values), and dropped `widget_background.xml`'s 1dp border stroke for a flatter look. Final on-device visual check still needs an owner Studio build, same as all Android work |
| **4.22.2** | **Sync stability fixes** (E2 — owner-reported "not stable yet," 2026-07-20). Found debugging a corrupted CouchDB shard on the dev machine, all three from the same root-cause family — the app trusting a stale identity instead of re-checking it. **E2**: the phone stored a fixed IP:port at pairing time and never revisited it, so any DHCP lease renewal or router reboot silently broke sync until a manual re-pair. Now persists the CouchDB server's own stable `uuid` (already broadcast over mDNS, already returned by pairing) and re-resolves it automatically on a "Cannot reach sync server" failure, updating the stored URL and retrying — no more manual re-pairing for an IP that just moved. **Dev/prod Tauri identity collision**: `cargo tauri dev` and a real installed build shared the exact same `sync-host.json` (port/credentials) despite normally running against different CouchDB data — a phone could end up silently talking to the wrong build's database. Debug builds now get their own config file and, if the usual per-target dev CouchDB copy is ever missing, their own isolated bootstrap copy instead of ever running live out of the one shared `vendor/couchdb-win` source directory. **Android debug/release storage collision**: a debug APK shared the same `applicationId` as a real install, overwriting/sharing its storage on every Android Studio test run — debug builds now install side by side via `applicationIdSuffix ".debug"` |
| **4.22.1** | **Maintenance pass** (eighth run, after v4.22.0 shipped, per the owner-set schedule in MAINTENANCE.md — first pass since v4.19.1, covering v4.20.0/v4.21.0/v4.22.0's changes). Found and fixed one real gap: `ChangelogView.svelte`'s "Clear all" button called the mutating `clearLogs()` with no try/catch/`showError()`, unlike every other mutation site (an audited invariant since v2.9.0) — fixed. Also added a Tauri window `minWidth`/`minHeight` (640×480) to `offlog-desktop`'s `tauri.conf.json`, since the window previously had no floor on how small it could be resized. Everything else checked out clean: no dead code, no new duplication, `npm audit` unchanged (same 4 dev-tooling-only advisories, none reachable from the shipped bundle), every `{@html}` site still correctly escaped/fixed-constant, no `eval`/`new Function`/stray `innerHTML=`, `offlog-desktop`'s `unsafe` blocks limited to the expected `TerminateJobObject` FFI calls, `pairing.rs` never logs credentials by value. One correction from the initial audit: `pouchdb` (the npm package, not `pouchdb-find`) looked unused in `src/` but is actually imported directly by `tests/setup.ts` as the real PouchDB stand-in during tests — left in place, removing it would have broken `npm test`. Disabled CSP (`security.csp: null` in `tauri.conf.json`) noted but deliberately not touched this pass — bundling it into the same pre-public-release security pass as C7 instead of a piecemeal maintenance fix, since enabling it needs real testing against the app's inline styles and dynamic-port sync fetches |
| **4.22.0** | **Week-start-day setting + full animation pass** (B47, B51 — owner-directed, 2026-07-20). **B47**: Settings → Appearance gets a Sunday/Monday "Week starts on" toggle, wired into Agenda's week grid and `DeadlinesView.svelte`'s "this week" grouping (both previously assumed a fixed Sunday start). Timezone, the other half of B47's original scope, deliberately not built — the app's local-time-only design is correct for a single-device personal task manager, see ROADMAP.md's B47 entry. **B51**: owner feedback that many actions "snapped" with no animation, reading as fake — an inventory pass found the real pattern was that things which *open* mostly animated already, but almost nothing that *closed* did, plus several everyday actions (checklist toggle, card create/delete/archive, the new v4.21.0 "⋯" menus) had no transition in either direction. New `src/lib/motion.ts` centralizes shared transition parameters (some panels/toasts needed *custom* transition functions, not Svelte's built-ins — those clobber an element's own `transform: translate(-50%,-50%)` centering mid-animation); every panel, dialog, popover, toast, card, and column that was missing open/close animation now has one. Also fixed, found in the same pass: `offlog-desktop`'s Tauri WebView was intercepting native OS drag-drop by default, silently breaking the Kanban board's HTML5 drag-and-drop on the PC app — `dragDropEnabled: false` in `tauri.conf.json` fixes it (no separate `offlog-desktop` build/sync step needed for the web-side changes here — `frontendDist` points straight at `offlog-app/dist`, so a normal `npm run build` updates both apps at once) |
